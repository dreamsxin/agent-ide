# Protocol

This package holds shared protocol documents and future generated client bindings.

The Rust runtime is the source of truth for trusted-boundary models. TypeScript bindings can be generated or mirrored from those models as the protocol stabilizes.

## Current Contract Surface

The V3 foundation currently exposes these protocol families:

- `workspace`: open workspace, list entries, read files, save files, and inspect Git state
- `command`: run a workspace-scoped command and return final output
- `execution`: track command and test run lifecycle state
- `runtime-event`: stream command output and runtime logs into the workbench
- `task`: discover common workspace tasks such as check and test commands
- `agent`: decompose user goals into reviewable implementation plans

## Command Protocol

`CommandRequest`

- `command`: shell command string executed relative to the trusted workspace root

`CommandResult`

- `command`: command string that was executed
- `success`: whether the process exited successfully
- `exit_code`: numeric exit code when available
- `stdout`: collected standard output
- `stderr`: collected standard error

`CommandStreamEvent`

- `stream`: `stdout` or `stderr`
- `line`: one emitted output line

The current streaming event is intentionally line-oriented. This keeps the first runtime event model simple while preserving enough structure for logs, tests, and future Agent task output.

## Task Protocol

`WorkspaceTask`

- `id`: stable task identifier, for example `rust.check`
- `label`: user-facing action label
- `command`: command that should run through the command protocol

Task discovery is owned by the Rust runtime because it inspects trusted workspace state. The client renders the discovered task list and sends selected task commands back through the command protocol.

## Runtime Event Protocol

`RuntimeLogEvent`

- `level`: `info`, `success`, or `error`
- `message`: human-readable runtime activity message

Runtime events are presentation-safe summaries. They should not become the source of truth for execution state; a dedicated execution model should be introduced before cancellation, parallel tasks, or Agent orchestration.

## Agent Protocol

`AgentTaskRequest`

- `goal`: user-authored task description to decompose

`AgentPlan`

- `id`: stable plan identifier
- `goal`: normalized task goal
- `summary`: plan summary for the review surface
- `source`: `local` or provider id used for planning
- `steps`: ordered implementation steps
- `verification`: recommended verification commands or checks

`AgentPlanStep`

- `id`: stable step identifier within the plan
- `title`: short step label
- `detail`: implementation guidance
- `status`: current step status, currently `pending`, `active`, or `done`
- `command`: optional command that can run through the shared execution model

The current Agent protocol is a local decomposition scaffold generated by the Rust runtime. A future provider-backed Agent should refine this plan and later execute steps through the shared execution model.

Verification items can be routed through the command execution protocol so the Agent planning surface and runtime execution surface stay connected.
Agent step commands are associated with runtime executions by execution id. When a step execution finishes successfully, the step can move to `done`; failed or cancelled runs keep the step actionable.

`AgentStepStatusRequest`

- `plan_id`: target plan id
- `step_id`: target step id
- `status`: next status, currently `pending`, `active`, or `done`

Step state is now updated through the Rust/Tauri boundary instead of being mutated only in client memory.
Agent plans are persisted under the workspace `.agent-ide/plans` directory as JSON documents so the latest plan can be restored after reopening the workspace.

`AgentPlanSummary`

- `id`: plan id
- `goal`: plan goal
- `step_count`: total number of steps
- `done_count`: completed step count

The workbench can list plan summaries and load a selected plan by id.

## Provider Protocol

`AgentProviderConfig`

- `provider`: provider id, currently `local` or `openai`
- `model`: provider model label
- `mode`: `offline` or `provider-backed`

`AgentProviderStatus`

- `provider`: active provider id
- `model`: active model label
- `configured`: whether the runtime can use this provider path today
- `secret_configured`: whether a runtime-owned provider secret exists
- `mode`: active planning mode
- `message`: presentation-safe status text

Provider configuration is persisted under `.agent-ide/provider.json`. Secrets must remain outside the client and should be handled by a future runtime-owned secret store.
The current runtime-owned secret scaffold stores provider secrets under `.agent-ide/secrets/provider.secret` and never returns secret material to the client.
Provider planning attempts write redacted logs under `.agent-ide/provider-logs`; these logs may include provider/model/goal/outcome metadata but must never include secret material.

## Execution Protocol

`ExecutionState`

- `id`: stable execution identifier for the current run
- `command`: command associated with the execution
- `status`: `idle`, `running`, `succeeded`, or `failed`
- `startedAt`: presentation timestamp for when the run started
- `finishedAt`: presentation timestamp for when the run finished, or `null`
- `exitCode`: process exit code when available, or `null`
- `outputCount`: count of streamed output lines observed by the workbench

The current implementation tracks a single foreground execution. The next runtime iteration should move execution identity into Rust so cancellation and parallel execution can target runtime-owned processes.

`ExecutionEvent`

- `started`: emitted when the runtime-owned execution id is available
- `finished`: emitted when the process exits with success state and exit code
- `cancelled`: emitted when cancellation is requested and the runtime kill command succeeds

`CommandStreamEvent` now carries `execution_id` so output can be associated with the active runtime-owned execution.

## Next Protocol Step

Before adding deeper Agent orchestration, complete the `Execution` protocol family:

- `ExecutionOutput`
- execution history
- parallel execution semantics

This will let command runs, tests, and Agent tasks share the same lifecycle model instead of each inventing separate status handling.
